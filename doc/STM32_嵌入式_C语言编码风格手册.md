# STM32 嵌入式 C 语言编码风格与工程规范手册

| 项目 | 说明 |
|------|------|
| 版本 | V1.4 |
| 适用范围 | 基于 **STM32 系列**（ARM Cortex-M、STM32 HAL/LL、CMSIS、裸机或带 RTOS）的 **C 语言** 固件 |
| 目的 | 在团队内形成**可读、可审、可测、可移植、少隐患** 的统一基线；与**行业常见实践**、**MISRA C:2012** 及 **CERT C** 中面向嵌入式/安全敏感代码的原则**可对照**；**本文不是 MISRA/ISO 的替代或完整合规判据** |

**阅读约定**：**必须** 表示强约束，**应** 表示在正常情况下遵守，**可** 表示可选。若客户或行业规范（车规、工控、医械等）有更高或冲突要求，**以客户规范为准**。

---

## 1. 标准与基线

### 1.1 语言标准

- **应** 使用 **ISO C99 或 C11 子集**；避免依赖实现定义或未定义行为（**必须** 用静态分析/编译器警告尽早暴露）。
- **不** 使用 C++ 与 C 混编，除非子工程单独约定。
- 嵌入式**可** 禁用**动态内存**在运行路径上的使用：若用 `malloc`，**应** 仅在启动期分配且**必须** 检查返回值并定义失败行为。

### 1.2 编译与静态分析

- **应** 将编译器警告开至 **高**，常见目标：
  - GCC/Clang: `-Wall -Wextra -Werror`（或**至少** 将**新增** 警告当错误处理；存量代码可**分阶段** 把 `-Werror` 打开）。
- **可** 采用 **MISRA C:2012** 检查工具（LDRA、PC-lint、Coverity 等）作为**合规模块** 项目的门禁之一。
- **可** 采用 **MISRA** 的 **“Guideline”** 思路：在安全性要求一般的项目中，选取子集**手工对照** 而非全文认证。

### 1.3 可移植宽度与定宽类型

- **应** 使用 `stdint.h` 中的定宽整型进行协议、寄存器位、硬件宽度：`uint8_t`、`uint16_t`、`uint32_t`、`int32_t` 等；**不** 假设 `int` 宽度。
- 布尔：若使用 `stdbool.h` 的 `true`/`false` — **可** 与项目统一**一种** 风格，避免与整型 0/1 混用不交代。

### 1.4 与 CMSIS、HAL/LL 的关系

- **不** 与 ST 外设名、HAL 名冲突；**业务层** 不直接散布裸寄存器地址字面量，**应** 经**封装**（`Bsp_` / `App_` 等 模块 源 文件 见 **§3**，文件 名 仍 可 `bsp_*.c`）或**驱动层** 访问，便于换脚与单测打桩（视项目要求）。

## 2. 项目与文件组织

### 2.1 目录

- 常见分层（**可** 按产品裁剪）：

```text
Core/Src, Core/Inc     — 应用/业务、main
Drivers/STM32*         — 厂商生成/ HAL 或 LL
Bsp/                   — 板级支持包（与原理图/板子绑定）
App/ 或  Module/       — 功能模块
Third_Party/           — 只读第三方（保留原始许可与版本）
Test/ 或  Unit/        — 单测/主机侧模拟
```

- **一个对外 API 的模块** = **一对外部可见头 `xxx.h` + 一个主实现 `xxx.c`**，内部实现细节用 **`xxx_internal.h` 或 `static` 限域**，避免“万能头”。

### 2.2 头文件

- **必须** 使用**头文件防重入** 宏，例如：

```c
#ifndef BSP_USART_H
#define BSP_USART_H

/* ... */

#endif /* BSP_USART_H */
```

- **应** 采用 `#include` 只依赖**最近必要** 头，避免在 `.h` 中塞满实现细节。
- **不** 在可多次包含的 `.h` 里定义**非内联/非 static** 对象或函数体（**例外**：`static inline` 在单头内联的少数场景需团队一致）。

### 2.3 单文件行宽与换行

- **应** 行宽 **≤ 100 或 120 字符**（二选一、项目统一），过长**应** 换行，优先在**操作符后** 断行。
- **应** 使用 **LF** 作为行尾（**跨平台**时）；Git 上可配置 `core.autocrlf` 与 **`.editorconfig` 一致**。

### 2.4 缩进

- **应** 全项目统一：**4 空格** 或 **1 个 Tab 展开为 4 空格**（**二选一、勿混**）；**不** 混用可见空格与制表符同一段落。

### 2.5 大括号

- **行业常见（K&R 或 Allman 二选其一）**：
  - **K&R**：`if (cond) {` 开括号**同行**；**可** 作为 HAL 同风格、节省行数。
  - **Allman**：`if (cond)\n{` 括号**独占**；**可** 提高**块** 可视性。  
- **应** 全项目**只选一种**；`else` 与**前一** `if` 的**闭括号** 同风格排列。

```c
/* 示例：K&R */
if (u == 0U) {
    return ERR_ZERO;
} else {
    v = f(u);
}
```

## 3. 命名（本团队约定，V1.4）

本版约定：**对外的/模块级函数、结构体/枚举的 `typedef` 类型名、本文件 `static` 函数** 使用 **大驼峰（Pascal）+ 下划线** 区分 **模块/子功能**；**枚举成员** 与 **预处理宏** 相同：**全大写+下划线**（下表，**以** `IRR_`、`BSP_` 等**前缀** 与 宏 在 人读 上 区分）。**`static` 变量** 为 **`s_`+小写+下划线**；**其余变量与形参** 为 **小写+下划线**；**类型名** **不** 加 **`_t`** 后缀。预处理器**宏/常量** 为 **全大写+下划线**。

### 3.0 总览表（**必须** 全项目统一）

| 对象 | 规则 | 正例 | 反例 |
|------|------|------|------|
| 公开/模块 **函数**；本文件 **`static` 函数** | **Pascal+下划** 分模块与动作；**每段** 内为 **PascalCase** | `Bsp_Relay_SetPump` `App_Irrigation_FsmTick` | `bsp_relay_set_pump` `pump_duty_set` |
| **结构体 / 枚举 的 `typedef` 名** | **Pascal+下划**；**不** 加 `_t` | `Bsp_Usart_Config` `Irr_FsmState` | `bsp_time_config_t` `Bsp_Time_Config_t` |
| **枚举成员** | 与 **宏 相同**：**全大写+下划线**；**建议** 以 **模块/模块缩写** 前缀 防 与 `#define` 重名，如 `IRR_`、`BSP_` | `IRR_FSM_STATE_IDLE` `IRR_FSM_STATE_WATER` | `Irr_FsmState_Idle`（Pascal+下，**勿** 用于 枚举 成员） |
| 全局/局部/**形参**（**非** `static` 存储类对象） | **小写+下划** | `pump_duty_permille` `zone_idx` | `PumpDuty` `PUMP_DUTY` |
| **本文件内 `static` 变量** | 前缀 **`s_`** + **小写+下划** 其余 | `s_line_buf` `s_rx_head` | `s_LineBuf` `sline_buf` |
| **`#define` 常量/宏** | **全大写+下划**；**风格** 与 上一行 **枚举成员** 一致，**以 名 前缀/上下文** 区分 含义 | `BSP_USART_RX_RING_SIZE` | 把 宏 写成 小写 与 本表 冲 突 |

- **不** 与 **ISO C 保留/实现** 的 **`__`、单下划开头** 的标识符 冲突。  
- **不** 把 **Pascal+下** 用于 **宏名** 或 **枚举成员**（**二者** 在 本 表 中 均为 全 大写+下 划）。  
- **全大写 枚举 成员 与 全大写 宏 同名** 时 属 **不同 名字 空间/语法**，但 仍 应 通过 **前缀 约定** 在 人 读 上 不 混 消（例：`IRR_FSM_STATE_IDLE` 与 `BSP_BOARD_REV`）。  
- **ST 厂商** 的 `HAL_` / `LL_` **不** 改为 本 风格，**不** 批量 重命名 生成 码。

### 3.1 大驼+下划 的 写法

- **下划线** 只 用于 较大 粒度 分界：`Bsp_` `Irr_` `App_` 以及 子域 `Relay_` `Usart_` `Fsm_`；**段数** 一般 **2～4**，避免 `A_B_C_D_E_F` 过长。  
- 每 **被下划 分开 的 段** 用 **Pascal** 拼 复合词，如 `SendBuffer`、`SetPump`、`Tick1ms`（`Tick`+`1ms` 的 数字+单位 组合 **全 项目 统一 一种 写法**）。  
- 函数 **谓词/查询** 可 用 `Is`/`Get` 段 名，如 `Bsp_Time_IsRdy` 或 `Bsp_Time_TickMsGet`（**动宾** 在 一 种 语言 里 统一 即可）。  
- **中断/HAL 回调** 的 **签名** 若 由 ST/启动 固 定 名 可 保持 厂商 名；**内部** 再 转 `Bsp_` 的 **短 封装**。

**示例**：

```c
void Bsp_Relay_SetPump( bool is_on );
void Bsp_Usart_SendBuffer( const uint8_t *data, size_t len );
static void Bsp_Usart_ParseLine( void );

typedef struct {
    uint32_t tick_ms;
} Bsp_Time_Config;

typedef enum {
    IRR_FSM_STATE_IDLE = 0,
    IRR_FSM_STATE_WATER
} Irr_FsmState;
```

**局部/形参（小写+下划）**及 **`static` 量**：

```c
void f( int zone_idx, uint32_t precharge_ms ) {
    uint16_t pump_duty_permille = 0U;
    static uint8_t s_line_buf[128U];
    static size_t  s_line_len;
    (void)precharge_ms; (void)zone_idx; (void)pump_duty_permille;
    (void)s_line_buf; (void)s_line_len;
}
```

### 3.2 源/头 **文件名**（**推荐**）

- **应** 仍 **小写+下划**：`bsp_relay.c`、`bsp_usart.h`，减少 **跨平台/大小写** 仓库 问题。  
- **防重入** 宏 可 `BSP_USART_H`，与 类型 `Bsp_` **二义** 无妨，**一表** 说明 即可。

### 3.3 变量 细则（小写+下划；`s_`+小写+下为 static）

- 单位 放 名 里：`precharge_ms`、`d_max_permille`；`i`/`j` 仅 极 短 循环 内。  
- 指针：后缀 `_p` 或 前缀 `p_`，**全项目 选 一种**。  
- 全局 非 `static` 仍 **可** 选 `g_`+小写+下 若 团队 要 强化；**本版** 已 用 `s_` 统一 **文件级 static**，`g_` 为**可选 扩展**。

### 3.4 宏、`#define` 常量、枚举成员、只读字符串

- 预处理**宏**及 **`#define` 的常量名** 为 **全大写+下划线**；**枚举成员** **同样** 为 **全大写+下划线**（**与** 宏 **书写风格** 一致，**以** 前缀/缩写 **区分 语义**）。`static const char` 只读串**可** `k_cmd_help`（`k_`+小写+下划）或 用 `#define` 大写串名，**二选一并成文**。  
- `static inline` 函数**名** 按 **3.0 表** 中**函数** 行（Pascal+下划），**勿** 与 **全大写 宏/枚举 成员** 在 **同一 可见域** 内 起 **完全 相同** 且 **易 混 的 短 名**。

### 3.5 硬件/外设/通道

- 在 **业务/类型** 里 优先 符号 名 如 `Valve_Z1` 作为 **Pascal+下 的 段 名** 出现 在 **结构体/枚举/函数** 中 时，**保持** 与 原理 图/手册 **注释** 对照；`PA2` 仅 在 **Bsp_** 的 **脚 映射 表** 一 处 维护。

## 4. 函数

### 4.1 长度与职责

- **应** 单函数**单一职责**；**应** 控制**认知长度**（常见指导 **&lt; 200 行**，硬上限**视工具** 或 **MISRA 局部规则**）。
- **不** 通过**过深** 嵌套表达逻辑；**应** 用**早返回**、子函数 减少圈复杂度。

### 4.2 形参与返回值

- **不** 依赖**形参** 的求值顺序（C 中未规定）。
- 错误码**应** **一致枚举** 或 项目约定的 `int32_t` 错误表；**0** 或 约定值表示成功，**必须** 文档化。
- **输出参数** 若可选，**应** 在文档说明 `NULL` 表示忽略。

### 4.3 禁止事项（嵌入式常见“坑”）

- **必须** 避免在**中断** 中调用**不可重入** 的**库**（如部分 `printf` 实现、复杂 `malloc` 路径）。
- **不** 在中断里**长时间** 关全局中断、**不** 在 ISR 里**打印大量字符** 以免丢字节或影响实时性；**可** 仅置**标志** 到主循处理。

### 4.4 `static` 与可见性

- 文件内辅助函数、变量**应** `static`，**缩小** 链接可见域。

## 5. 与硬件和并发相关的 C 使用

### 5.1 `volatile`

- **只** 对**硬件寄存器、由 ISR/其他线程 写的共享标志** 使用；**不** 滥用 `volatile` 代替**正确的同步/原子** 设计。
- **与主循通过标志通信**：主循读、IRQ 写时，**应** 有**内存屏障/屏障宏** 意识（Cortex-M 在多数简单标志场景下，**对易变量** 仍**建议** 以设计评审为准）。

### 5.2 位操作

- **应** 使用**定宽** 无符号，对位域的**可移植** 有要求的代码**避免** 依赖**结构体位域** 布局（MISRA 也限制位域的若干用法）；**可** 用 `uint32_t` + 掩码宏。

```c
#define REG_IE_POS   (8U)
#define REG_IE_MASK  (1UL << REG_IE_POS)
```

### 5.3 未使用

- 未用形参/变量：**可** `(void)name;` 或 `(void)argc;` 以通过 `-Wextra` 的**未用** 告警；**可** 使用编译器**属性/pragma** 抑制（**审慎**、局部）。

## 6. 注释与文档

### 6.1 何时写

- 公开 API 在 `.h` **应** 有**简短** 用/**前置条件**/**后效**/**可重入/线程与 IRQ 安全** 说明；实现若**非显式** 的算法**应** 有**行内** 或**块** 注。

### 6.2 风格

- 文件头**可** 含**版权/ SPDX**、**模块职责**、以及 Doxygen 的 **@file**、**@brief** 等标签。

```c
/**
 * @file    bsp_relay.c
 * @brief   继电器与泵/区阀的 GPIO 封装
 */
```

- 单行用 `/* ... */` 或 `//`（**C99**）；**多行** 用 `/*` 块；**不** 在注释里**嵌套** 易混淆的 `//`+`/*`。

### 6.3 禁止

- **不** 在版本控制系统里**保留** 大量**大段** 的**被注释死代码**；**应** 用分支或**删除+提交说明** 追溯。

## 7. 可测试性与可维护

- 纯**算法/协议** 层**可** 与**硬件** 分离，在 PC 上跑**单测**（或模拟 HAL）。
- **不** 将**业务分支** 与**延时忙等** 绑死，**应** 注入**时间** 抽象或**在测试** 中缩短**节拍**（若成本允许）。

## 8. 与常见标准/指南的对照关系（非合规声明）

- **MISRA C:2012**：面向**高可靠性** 车辆、轨交等；规则数量多，**分“Mandatory/Required/Advisory”**。**本文** 中的“须避免未定义行为、定宽整型、ISR 约束” 与**多数** Mandatory/Required 精神一致；**正式** MISRA 认证**应** 用**经认证** 工具+偏差报告**Deviation 流程**。
- **CERT C**：侧重**安全漏洞** 类 C 模式；在**有网络/USB 栈** 场景**应** 额外对照。
- **BARR-C 编码规范** 等**嵌入式** 书：强调**可预测**、**防御式** 编程，可与本文一并作为**内训** 材料。
